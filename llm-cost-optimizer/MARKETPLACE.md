Оригинал из коллекции alirezarezvani/claude-skills, адаптирован и локализован командой COMANDOS.

## 😩 Знакомая боль?

- **«Расходы на API выросли в 10 раз, но я не вижу за что»** — нет логирования по фичам, всё в чёрном ящике. Даже не знаешь, какой эндпоинт жрёт больше.
- **«Мы пакуем все запросы в самую мощную модель»** — потому что просто и надёжно. Но 80% запросов — это классификации и формотирование, которые справляет дешёвая модель в миллион раз быстрее.
- **«Система-промпт 3000 токенов, отправляю на каждый запрос»** — никогда не слышал про prompt caching. Платишь за одно и то же 1000 раз в день.
- **«Запросы вариативные, кэш не помогает»** — а может, это семантическое кэширование? Половина запросов семантически идентичны, просто сформулированы по-разному.
- **«Спешим, мониторим потом»** — потом — это когда деньги уже потрачены. Или когда spend spike просрочится на недельку.

Корень один: **расходы на LLM — это инженерный процесс, а не рок-н-ролл.** Надо мерить, логировать, проектировать маршруты и мониторить как database queries.

## 💸 Что делает этот навык

Вводит методологию cost engineering в твой LLM-стек. Не просто «расходы плохие», а конкретный лист дорог снижения:

- **Проводит cost audit** — берёт твою архитектуру, логирует токены по каждой фиче-модели паре, находит Парето (обычно 2–3 эндпоинта жрут 80% бюджета).
- **Классифицирует запросы по сложности** — выделяет, какие просто classification (отправить на Haiku/GPT-4o-mini), какие reasoning (нужен Opus/o3). Наивный routing сразу режет траффик и расходы в 5–10x на простых задачах.
- **Диагностирует prompt caching** — если система-промпт >2000 токенов или текут документы-контекст, это высокоценный кэш-кандидат. И вычисляет hit rate.
- **Проектирует семантическое кэширование** — когда точное совпадение редко, но смысл повторяется. Threshold: cosine similarity >0.95.
- **Аудит на token waste** — max_tokens не установлен? Это утечка. Промпт можно сжать на 20–30%? Можно (с осторожностью).
- **Строит мониторинг** — cost per feature, cost per user tier, anomaly alerts. Бюджет-конверты по фичам. Graceful degradation: бюджет исчерпан → switch на дешёвую модель → cached response → async queue.

На выходе **не просто список советов**, а:
- Per-feature spend breakdown (барчарт: что жрёт что)
- Top 3 optimization targets с прогнозом экономии (реалистичным)
- Cost-efficiency scorecard 0–100
- Routing decision tree (для каких задач какая модель)
- Caching strategy (что, как долго, какой hit rate ожидать)

## 🔄 Как это работает — три режима

| Режим | Когда | Что происходит |
|-------|-------|---|
| **Cost Audit** | Расходы есть, но пробелов в видимости | Инструментация логирования, поиск 20% затрат на 80% бюджета, классификация по сложности |
| **Optimize Existing** | Архитектура известна, надо режем | 1) Model routing (60–80% экономия) 2) Prompt caching (40–90%) 3) Output length control 4) Prompt compression 5) Semantic cache 6) Request batching |
| **Design Cost-Efficient** | Запускаем новый AI-фичу с нуля | Бюджет-конверты, routing layer, tier model access by user tier, observability dashboard, graceful degradation |

Каждый режим = пошаговая методология. Не прыгаешь сразу на prompt compression, если нет логирования.

## 🎯 Когда включать

- **Перед запуском любого AI-фичи** — wire cost controls in, не после.
- **Когда расходы выросли** — cost audit это твой первый шаг.
- **Когда моноулитура по одной дорогой модели** — это сигнал пересмотреть архитектуру.
- **Когда видишь system prompt >2000 токенов на каждый запрос** — кэширование вот-вот сэкономит 50% бюджета.
- Не нужен для: tweaking готовой системы в пределах 10% (там просто логирование достаточно).

## 📦 Что на выходе

- **Cost breakdown таблица** — spend per feature, per model, per user tier, week-over-week trend.
- **Routing decision tree** — для каких задач какая модель, с estimated cost delta.
- **Caching strategy document** — what to cache, key design, expected hit rate, implementation pattern.
- **Prompt optimization audit** — token-by-token analysis, compression suggestions, before/after token count.
- **Scorecard 0–100** — текущее состояние, приоритизированный список фиксов, projected monthly savings.
