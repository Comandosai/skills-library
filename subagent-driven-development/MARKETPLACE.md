## 📄 ДЕТАЛЬНАЯ СТРАНИЦА

# Разработка через субагентов

> Выполняй план задачами — на каждую свежий субагент и двойная проверка: сначала по ТЗ, потом по качеству.

Оригинал из коллекции Superpowers (obra), адаптирован и локализован командой COMANDOS.

## 😩 Знакомая боль?

- **«Агент тащит весь проект в одной сессии»** — к пятой задаче контекст замусорен, он забывает, что делал в начале, и начинает противоречить сам себе.
- **«Сделал — а оно не по ТЗ»** — половину требований пропустил, зато добавил то, чего никто не просил.
- **«Код вроде работает, но качество — стыдно показать»** — магические числа, дубли, ноль проверок.
- **«Проверять каждую задачу вручную дорого»** — а без проверки в проект утекает брак.

Корень один: **один агент с одним замусоренным контекстом не вытягивает и реализацию, и контроль качества сразу.** Этот навык разводит роли по конвейеру.

## ⚡ Что делает этот навык

Превращает исполнение плана в конвейер из свежих субагентов с обязательным двойным контролем:

- **Свежий субагент на каждую задачу.** Чистый контекст, полный текст задачи на входе — никакой мешанины из предыдущих шагов.
- **Субагент делает всё сам:** реализует, тестирует, коммитит, проводит саморевью — и только потом отдаёт дальше.
- **Двухступенчатое ревью.** Сначала ревьюер сверяет результат с ТЗ: всё ли сделано, не добавлено ли лишнего. И только при зелёном свете — второй ревьюер смотрит качество кода.
- **Цикл правок.** Нашли проблему — тот же субагент чинит, ревью повторяется, пока не пройдёт. «И так сойдёт» здесь не принимается.
- **Без передачи человеку между задачами.** Ты отвечаешь на вопросы субагента, если он их задаёт, — остальное конвейер ведёт сам, в одной сессии.

## 🔄 Как это работает

| Шаг | Что происходит | Зачем |
|------|----------------|-------|
| **1. Разобрать план** | Вытащить все задачи с полным текстом, занести в список | Чтобы давать субагенту контекст, а не ссылку на файл |
| **2. Реализовать** | Свежий субагент: код, тесты, коммит, саморевью | Чистый контекст на каждую задачу |
| **3. Сверить с ТЗ** | Ревьюер проверяет соответствие спецификации | Чтобы не было «недоделано» и «лишнего» |
| **4. Проверить качество** | Второй ревьюер смотрит код — только после зелёного по ТЗ | Чтобы реализация была не только полной, но и чистой |
| **5. Цикл правок** | Есть замечания → тот же агент чинит → ревью снова | Чтобы фиксы действительно закрывали проблему |

> 💡 Главный приём — **строгий порядок проверок: сначала ТЗ, потом качество**. Нет смысла вылизывать код, который делает не то. Сперва — «то ли построили», затем — «хорошо ли построили».

## 🎯 Когда включать

- Есть готовый план/спека, и задачи в нём в основном независимы.
- Хочется довести проект до конца в одной сессии, без ручной передачи между задачами.
- Важно, чтобы каждый кусок прошёл сверку с ТЗ и проверку качества, а не просто «вроде работает».

Не нужен, когда плана ещё нет (сначала — брейншторм и план), задачи жёстко связаны друг с другом, или это разовая мелкая правка.

## 📦 Что на выходе

- Реализованный план, задача за задачей, с коммитами по ходу.
- Каждый кусок прошёл сверку с ТЗ и проверку качества кода.
- Зафиксированные циклы правок: что было не так и как починили.
- Финальное ревью всей реализации перед завершением ветки.
