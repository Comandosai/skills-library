# RAG: база знаний

> Собирает бота и поиск, которые отвечают по твоим документам — а не выдумывают.

Навык из коллекции «Сообщество Claude Code Skills», адаптирован и локализован командой COMANDOS.

## 😩 Знакомая боль?

- **«Просишь бота ответить про свой продукт — а он сочиняет»** — звучит уверенно, но факты выдуманы, ссылки битые, цифры из головы.
- **«Закинул базу в промпт — не влезает»** — документов на сотни страниц, в контекст модели всё это не помещается физически.
- **«Сделал поиск по эмбеддингам — выдаёт мусор»** — порезал текст по 500 символов, и теперь в ответах обрывки фраз без смысла.
- **«Бот отвечает не по тем кускам»** — нужный абзац в базе есть, но поиск его не находит, а тащит что-то соседнее.
- **«Обновил документы — бот отвечает по-старому»** — данные поменялись, а эмбеддинги остались прежними, и никто за этим не следит.

Корень один: **между твоими данными и моделью нет нормального слоя поиска.** Этот навык его и строит — чтобы в ответ попадали реальные факты из твоих документов, а не правдоподобные выдумки.

## 📚 Что делает этот навык

Превращает агента в инженера RAG (Retrieval-Augmented Generation), который **строит систему ответов поверх твоих данных** — а не очередную болталку:

- **Грамотно нарезает документы** — по смыслу, а не вслепую по символам. С перекрытием (overlap), чтобы не рвать мысль на полуслове.
- **Считает эмбеддинги и кладёт в векторную базу** — единой моделью для документов и запросов (рассинхрон моделей — классическая поломка).
- **Настраивает гибридный поиск** — вектор (смысл) + ключевые слова (точные термины, артикулы, имена), чтобы находить и по сути, и по точному совпадению.
- **Переранжирует найденное** — отбирает действительно релевантные куски, прежде чем отдать их модели в контекст.
- **Честно скажет, когда RAG не нужен** — если задача решается проще, навык не будет городить векторную базу ради галочки.

На выходе — **рабочий слой поиска по твоим данным**: бот-консьерж, внутренний помощник по регламентам, поиск по базе знаний. Код фреймворк-агностичен — ляжет на твой стек и любую векторную БД.

## 🔄 Как это работает

| Этап | Что происходит | Зачем |
|------|----------------|-------|
| **1. Нарезка (chunking)** | Документы дробятся на смысловые куски с перекрытием | Плохие чанки = плохой поиск; это фундамент качества |
| **2. Эмбеддинги** | Каждый кусок переводится в вектор одной моделью | Чтобы запрос и документы жили в одном пространстве смыслов |
| **3. Векторная база** | Векторы складываются в хранилище под быстрый поиск | Чтобы искать по миллионам кусков за миллисекунды |
| **4. Поиск + reranking** | Гибридный поиск находит кандидатов, реранкер отбирает лучшие | Чтобы в контекст модели попало именно то, что отвечает на вопрос |

> 📚 Главный принцип — **RAG это не просто «вектор-поиск», а доставка нужной информации модели в нужный момент.** Качество ответа решается на этапе нарезки и поиска, а не на этапе генерации.

## 🎯 Когда включать

- Когда нужен бот/помощник, отвечающий **по твоим документам**: регламенты, база курса, поддержка, внутренние доки.
- Когда данных слишком много, чтобы запихнуть их в промпт целиком.
- Когда уже сделал «чанк-и-эмбеддинг наивно», а поиск выдаёт мусор и хочется сделать правильно.
- Когда важно, чтобы ответы были **с опорой на факты**, а не на фантазию модели.

Не нужен для задач, где данные целиком влезают в контекст или ответ не зависит от твоей базы — там RAG только добавит сложности и задержку.

## 📦 Что на выходе

- Рабочий пайплайн RAG: нарезка → эмбеддинги → векторная база → поиск с переранжированием.
- Бот/поиск, который **отвечает по твоим данным**, а не выдумывает.
- Код под твой стек и любую векторную БД — фреймворк-агностичный, без жёсткой привязки.
- Понимание, где у тебя «острые углы» (рассинхрон эмбеддингов, латентность, качество чанков) и как их закрыть.
