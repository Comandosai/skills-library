# MCP-конструктор

> Строит MCP-серверы — мостики, через которые ИИ дёргает твой CRM, API и любые внешние сервисы.

Навык из сообщества Claude Code Skills, адаптирован и локализован командой COMANDOS.

## 😩 Знакомая боль?

- **«Хочу, чтобы агент работал с моей CRM»** — а он не умеет, потому что у него нет моста к твоим данным.
- **«Есть сервис с API, но как его дать агенту?»** — копировать ответы руками в чат — это не автоматизация.
- **«Накидал интеграцию на коленке»** — агент путается в инструментах, получает простыни данных и тонет в них.
- **«Сделал под Claude, а теперь нужно и в Cursor»** — и снова переписывать всё с нуля под другой клиент.
- **«Сервер вроде работает, но агент им не пользуется»** — потому что инструменты названы криво и описаны непонятно.

Корень один: **между ИИ и внешним сервисом нет нормального моста.** Этот навык строит такой мост по стандарту, который понимают сразу несколько ИИ-клиентов.

## 🔌 Что делает этот навык

Включает в агенте режим **инженера интеграций**. Ты формулируешь задачу человеческим языком — он сам доводит её до рабочего MCP-сервера:

- **Исследует API сервиса** — какие есть эндпоинты, как авторизация, какие модели данных. Web-поиск и чтение доков на нём.
- **Проектирует инструменты** — не просто оборачивает каждый эндпоинт, а думает, чем агенту будет удобно пользоваться: понятные имена с префиксами, аккуратные схемы входа/выхода, пагинация.
- **Пишет сервер** на твоём стеке — Python (FastMCP) или Node/TypeScript (MCP SDK). С Zod/Pydantic-схемами, обработкой ошибок и аннотациями безопасности.
- **Собирает и тестирует** — `npm run build` или проверка синтаксиса, прогон через MCP Inspector, чтобы убедиться, что инструменты реально вызываются.
- **Делает набор проверок** — 10 реалистичных сценариев, на которых видно, что агент действительно решает задачи через твой сервер.

На выходе — **рабочая интеграция**, через которую ИИ дёргает внешний сервис. И поскольку MCP — открытый стандарт, один сервер понимают Claude, Cursor, Codex и Gemini.

## 🔄 Как это работает — четыре фазы

| Фаза | Что происходит | Зачем |
|------|----------------|-------|
| **1. Исследование и план** | Разбор API сервиса, изучение протокола MCP и SDK, список инструментов | Чтобы строить под реальный API, а не под догадки |
| **2. Реализация** | API-клиент, схемы, инструменты, обработка ошибок, пагинация | Чтобы получился сервер, которым агенту удобно пользоваться |
| **3. Сборка и тест** | Компиляция, проверка через MCP Inspector | Чтобы убедиться: инструменты вызываются и отдают то, что нужно |
| **4. Набор проверок** | 10 реалистичных сценариев с проверяемыми ответами | Чтобы видеть, что интеграция реально работает, а не «вроде запустилась» |

> 💡 Главный приём — **проектировать под агента, а не под API**. Понятные имена инструментов, сфокусированные ответы и честные ошибки решают, будет агент пользоваться сервером или будет в нём путаться.

## 🎯 Когда включать

- Когда нужно подключить агента к своей CRM, базе, платёжке или любому сервису с API.
- Когда у сервиса есть API, но нет готового MCP-сервера — и надо сделать свой.
- Когда интеграция должна работать сразу в нескольких клиентах (Claude, Cursor, Codex, Gemini).
- Когда самодельный сервер уже есть, но агент им не пользуется — пора переделать по гайдлайнам.

Не нужен для разовых запросов к API «дёрнуть один раз и забыть» — там проще обычный скрипт.

## 📦 Что на выходе

- Рабочий MCP-сервер на Python (FastMCP) или Node/TypeScript (MCP SDK) в твоём проекте.
- Набор инструментов с понятными именами, схемами входа/выхода и аннотациями безопасности.
- Сервер собран и проверен через MCP Inspector.
- Файл с 10 проверочными сценариями, по которым видно, что интеграция реально решает задачи.
